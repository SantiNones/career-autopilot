## Engineering Review

### 2. Which areas still rely on manual testing?

| Area | Reason | Risk Level |
|------|--------|------------|
| **LLM requirement extraction** (V3/V4 real flow) | Requires OpenAI calls, non-deterministic | MEDIUM |
| **Evidence Engine** | Requires OpenAI + DB (Candidate Intelligence) | MEDIUM |
| **OpenAI Materials** (openaiMaterials.ts) | Full AI generation path not tested | MEDIUM |
| **Discovery provider integration** | Requires live API calls to Adzuna/etc. | LOW |
| **Frontend components** | No component tests yet | LOW |
| **Database operations** | Integration tests not in scope | LOW |

### 3. Which benchmark gives the highest confidence?

**Capability Matching (22 tests)** — This suite covers the entire deterministic core: taxonomy integrity, exact/adjacent/transferable matching, confidence ranking, evidence tier resolution, and regression protection. Since the capability system is the foundation of V4 scoring, protecting it provides the highest systemic confidence.

### 4. Which subsystem is hardest to benchmark?

**Evidence Engine** — It requires:
1. A real Candidate Intelligence record (generated by OpenAI)
2. Resume parsing output
3. OpenAI calls for evidence extraction

The benchmark path requires either stored golden outputs (fixture snapshots from a real run) or an opt-in AI benchmark command. This is the first candidate for `npm run benchmarks:ai`.

### 5. What benchmark should be added before Discovery V2?

**Discovery Quality Baseline** — Before redesigning discovery, capture:
- Current top-20 precision for a fixed set of queries
- Score distribution histogram (what % of results are APPLY vs SKIP)
- Provider response fixture snapshots
- Deduplication correctness on known duplicate pairs
- Recommendation ranking stability

This provides a "before" snapshot to objectively measure improvement.

### 6. What benchmark should be added before Python migration?

**API Contract Verification** — Before migrating any subsystem to Python:
- Snapshot current API route inputs/outputs as JSON fixtures
- Verify the Python implementation produces identical outputs for the same inputs
- Add latency benchmarks (response time must not regress)
- Contract tests for every endpoint that will be migrated

This ensures the Python backend is a drop-in replacement.

### 7. If another engineer joins tomorrow, can they safely refactor the codebase using these benchmarks?

**Yes, for the deterministic subsystems** (scoring, capability matching, material templates, ingestion). They can:

1. Run `npm run benchmarks` before any change
2. Make their refactor
3. Run `npm run benchmarks` after
4. If all 89 pass → safe to merge

**Partial coverage gap:** The LLM-dependent paths (evidence engine, AI materials, real V4 with OpenAI requirement extraction) are not yet benchmarked. An engineer refactoring those would need manual verification or the future `benchmarks:ai` command.
